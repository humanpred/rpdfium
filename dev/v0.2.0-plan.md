# pdfium 0.2.0 — Mutation, authoring, and form-filling plan

Status: draft (post-0.1.0). This document accompanies, but does not yet
supersede, the post-0.1.0 candidate table in
[`r-pdf-ecosystem-survey.md`](r-pdf-ecosystem-survey.md). Decisions
captured here will be promoted to numbered ADRs under
[`decisions/`](decisions/) when work begins.

## 0. Reader / writer correspondence

The 0.1.0 read surface was deliberately designed for write-symmetry:
every tibble that a reader returns should be editable as an ordinary
R data frame and handed back to a writer with minimal reshaping. The
companion document [`reader-writer-audit.md`](reader-writer-audit.md)
walks every reader against the writers planned below. The headline:

* **Identity columns.** Every reader emits an `xxx_index` 1-based
  identity column (annotation_index, field_index, link_index,
  bookmark_index, mark_index, element_index, attachment_index,
  obj_index). The writer counterpart accepts the same index plus the
  implicit doc/page handle to locate the object.
* **Native PDF coordinate space.** All geometry is in PDF
  user-space points (`bounds_left`, `bounds_bottom`, `bounds_right`,
  `bounds_top` ordering). Writers consume the same convention; the
  C shim translates to PDFium's `FS_RECTF`.
* **List-columns for variable geometry.** Quad points, vertices,
  ink paths, dash arrays, option labels, marked-content params,
  structure attributes — every variable-length payload is already a
  list-column of matrices / vectors / named lists. Writers accept
  the same shape.
* **Codes and names coexist.** Annotation subtype, action type, fill
  mode, line cap / join, render mode, dest view all come back as
  readable strings *and* (where it matters) integer enum codes.
  Writers take either.
* **Normalized numeric ranges.** Colors are 0..1 doubles, matrices
  are 3x2, opacities are 0..1. PDFium's 0..255 / float32 / FS_*
  encodings stay inside the C layer.

These principles are locked in for v0.1.0; v0.2.0 writers SHOULD NOT
ask v0.1.0 readers to change shape.

### 0.1 Reader → writer round-trip contract

For every reader-writer pair below, the round-trip contract is:

> Read it. Modify a single cell (or list-column element). Hand the
> changed row (or its index + the new value) to the writer.
> `pdf_save()`. Re-open. The modification persists.

| Reader (column / list-column) | Writer setter signature |

Read-only by upstream constraint (PDFium does NOT expose setters):

* `pdf_bookmarks()` — outline tree mutation only via direct dict
  manipulation. Defer to v0.3+ or address with a high-level
  `pdf_bookmarks_replace(doc, df)` that rebuilds `/Outlines`.
* `pdf_signatures()` — signature creation is intentionally not in
  PDFium's public API.
* `pdf_structure_tree()` — `/StructTreeRoot` mutation needs raw
  dict access; defer indefinitely.
* `pdf_page_thumbnail()` — no setter in PDFium's public API.
* `pdf_text_chars()` (per-character) — PDFium has no per-glyph edit
  primitive; mutation routes through the parent text page object.
* `pdf_doc_security()`, `pdf_doc_permissions()`, `pdf_doc_xref_valid()`,
  `pdf_doc_trailer_ends()` — read-only by design; encryption changes
  are deferred.

## 1. Goal

`pdfium` 0.1.0 ships a comprehensive *read* surface. 0.2.0 adds a
deliberate, well-scoped *write* surface, in three buckets that map
directly to user demand visible in the R ecosystem:

1. **Form filling.** Read AcroForm widget values is in 0.1.0. The
   missing half — *setting* a widget's value, saving the result —
   is the most-requested missing feature when comparing to
   `staplr` (which requires a JRE).
2. **Document mutation** in the structural sense: rotate pages,
   delete pages, reorder pages, merge documents. `qpdf` already
   covers this on CRAN. We add it not to compete with `qpdf` but
   so users don't need to jump between two packages for end-to-end
   workflows. (`qpdf` remains the right choice for compress /
   linearise / split.)
3. **Annotation authoring.** Create highlights, sticky notes, and
   link annotations. The use case is markup tooling — programmatic
   commentary on PDFs — which has no R precedent today.

Non-goals for 0.2.0:

- Page-object authoring (drawing new paths / text / images via
  `FPDFPageObj_CreateNewPath`, `_NewTextObj`, `_NewImageObj`).
  This is a PDF *authoring* API; users who want to draw PDFs
  programmatically should reach for `grDevices::cairo_pdf` or
  `ggplot2` → ggsave. Reasonable 0.3+ feature if there is demand.
- Tagged-PDF authoring. Read in 0.1.0; write is a research-grade
  problem.
- Digital-signature creation (PKCS#7 signing). Better as a
  companion package (`pdfium.signatures`) that links OpenSSL.
- JavaScript / form-action mutation.

## 2. Architectural decisions to make before any code lands

The 0.1.0 memory model assumes documents are effectively immutable
after open. Mutation breaks several invariants we've leaned on; each
needs a deliberate decision (and an ADR) before code lands.

### 2.1 In-place vs. open/save lifecycle

PDFium's mutation API mutates the in-memory `FPDF_DOCUMENT`; persisting
to disk requires `FPDF_SaveAsCopy` (or `FPDF_SaveWithVersion`) writing
through a `FPDF_FILEWRITE` callback. Three options for the R surface:

| Option | Shape | Pros | Cons |
|---|---|---|---|
| A: explicit save | `pdf_open(path) → modify → pdf_save(doc, out)` | Matches `pdftools`/`qpdf` convention; lets callers chain edits | Two doc handles in memory if `out != in` |
| B: in-place edit | `pdf_edit(in, out, function(doc) ...)` | Single transaction; can't leak modified state | More opinionated; restricts ad-hoc edits |
| C: side-effect functions | `pdf_rotate_page(path, page, degrees, out = path)` | Matches `qpdf::pdf_rotate_pages` | Heavy per-call overhead; no editing graph |

**Recommendation: option A.** Aligns with how users already think
about R objects (open, modify, save). The doc handle remains the
unit of work. `pdf_save()` becomes its own export; an `out` argument
on `pdf_close()` is *not* the right shape (we'd be overloading a
release operation).

ADR-010 (proposed): "Mutation lifecycle: explicit pdf_save()".

### 2.2 Read-write document flag

Many PDFium mutators (e.g. `FPDFAnnot_SetStringValue` once an
`FPDF_FORMHANDLE` is attached) silently no-op or behave unpredictably
on documents not opened for editing. We should:

- Add a `readwrite = FALSE` default to `pdf_open()`.
- Track the flag on the S3 object so every mutator can refuse with a
  clear message ("Document opened read-only; reopen with
  `pdf_open(..., readwrite = TRUE)`").
- Internally this is just a slot on `pdfium_doc`; it doesn't require
  any PDFium-level capability.

This is also a hedge against accidental edits.

### 2.3 Form-fill environment lifetime

AcroForm interaction requires `FPDFDOC_InitFormFillEnvironment`,
which creates an `FPDF_FORMHANDLE` tied to the document. The handle
must outlive every mutation, then be explicitly released via
`FPDFDOC_ExitFormFillEnvironment` before document close.

Two approaches:

| Option | Shape | Pros | Cons |
|---|---|---|---|
| Lazy + cached | Spin up on first form mutation; cache on doc; release on `pdf_close` | Read-only users don't pay the cost | More state to track; ordering is subtle |
| Eager when `readwrite = TRUE` | Init at `pdf_open(readwrite = TRUE)`; release at `pdf_close` | Simpler lifecycle | Read-only edits of forms (rare but valid) need to spin one anyway |

**Recommendation: lazy + cached**, with the cache slot on the
`pdfium_doc` external pointer's `prot` field. Documented in the
architecture vignette.

ADR-011 (proposed): "Form-fill environment lifetime: lazy cache".

### 2.4 Atomic save semantics

PDFium's `FPDF_SaveAsCopy` does not write atomically; the
`FPDF_FILEWRITE` callback receives byte chunks. If the user passes a
real file path and the save aborts mid-write, the existing file is
corrupted. We should:

- Always save to a tempfile in the destination's directory and rename
  atomically on success. (R's `file.rename()` is atomic on the same
  filesystem.)
- Document this behaviour explicitly.

Not a separate ADR; handled inside `pdf_save()`.

### 2.5 Validation / round-trip testing

PDF mutators are notoriously easy to ship with silent corruption
(invalid xref tables, dangling object references, etc.). For every
mutation feature we should:

- Have a round-trip test: open → mutate → save → re-open → assert
  the mutation took.
- Optionally: validate the saved file with `qpdf --check` if
  available (skip on CRAN).

|---|---|

Once 0.1.0 lands on CRAN:

1. **ADRs 010–013** drafted and accepted (no code yet).
2. **Save / open-readwrite plumbing** lands first — every other

Estimated effort: ~6 working weeks of one engineer, plus review.

## 7. Risks and open questions

- **Document handle lifetime after `FPDF_ImportPagesByIndex`.**
  Importing pages between two documents creates references; closing
  the source before the destination may leave dangling pointers
  inside the destination. Verify experimentally before shipping
  `pdf_merge`.
- **PDFium's annotation appearance stream regeneration.** Setting
  highlight quadpoints does not always trigger PDFium to regenerate
  the appearance stream; the saved PDF may show the annotation in
  Acrobat but not in viewers that respect the AP entry strictly.
  Investigate `FPDFAnnot_SetAP` requirements.
- **AcroForm field flags** (read-only, required, no-export) — are
  they part of `pdf_form_fields()` output today? If not, add to the
  0.1.0 read pass before locking the column set.
- **Atomic-save behaviour on Windows.** R's `file.rename()` is not
  atomic across volumes; need to write the tempfile to the same
  directory as the destination. Document and test.
- **Vector-geometry round-trip via list-columns.** The 0.1.0
  readers surface quad points / vertices / ink paths / dash arrays
  / marks-params as list-columns. The writer functions that consume
  them MUST tolerate `NULL` (no change) AND empty matrices /
  vectors (clear-then-set semantics). Test for both paths
  explicitly per feature.
- **`/Vertices` and `/InkList` setters.** PDFium exposes
  `FPDFAnnot_AddInkStroke` and `_RemoveInkList` for ink, but no
  public setter for `/Vertices` on line / polygon / polyline. The
  writer for `pdf_annot_set_vertices()` will likely require
  falling back to direct dict access (which PDFium *does* allow via
  `FPDFAnnot_SetStringValue`-style key writes but not for arrays).
  Investigate experimentally; document the gap in `pdf_annotations()`
  if no clean path exists.
- **Bookmark mutation.** `pdf_bookmarks()` reader is already
  round-trippable in tibble shape, but PDFium has no public outline
  mutators. A future `pdf_bookmarks_replace(doc, df)` will need
  raw dict manipulation; not in 0.2.0.

## 8.1 Refresh: deferrals trimmed during the un-deferral pass

The original Tier-3 deferrals list (glyph paths, per-char font
info, render-with-matrix, generic annot probes, struct-tree
navigation extras, focusable subtypes) all landed in v0.1.0 after
a follow-up audit pass driven by the "challenging character
mapping" use case. See
[`reader-writer-audit.md`](reader-writer-audit.md) §"Tier 3
readers landed in 0.1.0" for the full list.

The deferrals that **remain** for v0.2.0+ are exclusively:

1. Mutation / writer surface (this entire plan).
2. `FPDFAnnot_GetObject` — wraps an embedded child page-object;
   needs an S3 class for "annotation-owned page object" that fits
   better with annotation authoring.
3. Streaming readers (`FPDFAvail_*`) — out of scope for the local-
   file workflow this package targets. Would only land if a future
   HTTP-backed wrapper appears.
4. Viewer-UI internals (`IsSupportedSubtype`, default TTF maps,
   custom-palette progressive rendering) — niche enough to keep
   off the CRAN-grade surface.

## 9. Reader-side leftovers identified during the audit

These small read-side adjustments landed *with* the audit (they're
in 0.1.0) and inform the v0.2.0 writers:

* `pdf_text_runs()` column `text_index` → `obj_index` rename, since
  it always indexed page objects and the name collided with the
  *extractable*-text offset on `pdf_text_chars()`.
* `pdf_annotations()` gains `subtype_code` integer column so writers
  taking `FPDF_ANNOT_*` enums round-trip without a name → code

* `pdf_form_fields()` gains `export_value` (distinct from `value`),
  `control_count` / `control_index` (for radio-button groups),
  `is_option_selected` list-column.
* `pdf_page_links()`, `pdf_link_at_point()`, `pdf_page_actions()`,
  `pdf_bookmarks()`, `pdf_named_dests()` all gain `dest_view` +

* `pdf_obj_marks()` and `pdf_structure_tree()$attributes` surface
  PDFium's typed parameter / attribute systems with R-typed values
  so v0.2.0 setters can take the same `list(...)` shape back.
