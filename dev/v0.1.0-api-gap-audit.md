# PDFium API gap audit at v0.1.0 release

A snapshot of which PDFium public symbols remain unwrapped by
`pdfium` as of the v0.1.0 CRAN candidate. Refresh by re-running the
methodology at the bottom of this file.

## Provenance

- Audit date: 2026-05-21.
- PDFium pin: `chromium/7202` (see `tools/pdfium-version.txt`).
- Method: grep every `FPDF_EXPORT` declaration in
  `inst/include/fpdf_*.h`, cross-reference against `src/*.cpp`,
  count the un-referenced symbols.
- Coverage at v0.1.0: **~258 / 399 exported symbols wrapped (≈64%)**.

The remaining ~141 unwrapped symbols are concentrated in three
groups, scored below for v0.2.0 priority.

## High-value v0.2.0 candidates

Symbols that close gaps in the existing v0.1.0 surface — they
either extend a reader to a new column, or add a writer for an
already-readable attribute.

### Document metadata writers (3-symbol upstream gap)

PDFium has no public `FPDF_SetMetaText` — only `FPDF_GetMetaText`.
This is the single most-requested writer gap from the v0.1.0 user
survey. The reader (`pdf_doc_info()`, `pdf_doc_meta()`) is solid;
the writer requires either an upstream patch (same shape as the
already-drafted `FPDFTextObj_SetFontSize` and
`FPDFAnnot_AppendOption` patches) or going through `FPDFCatalog_*`
which doesn't exist publicly either.

**Action**: draft `dev/upstream-patches/pdfium-FPDF_SetMetaText.patch`
as the v0.2.0 lead patch.

### Bookmark / outline authoring

Bookmarks read fine (`pdf_doc_bookmarks()`,
`pdf_doc_bookmark_find()`). Authoring (create, retitle, reparent)
needs PDFium's `FPDFBookmark_GetCount` (one of the unwrapped 6 in
`fpdf_doc.h`) plus a writer family that doesn't ship yet. Same
upstream-patch story as doc metadata.

### Action / destination details

Two symbols in `fpdf_doc.h` would let us extract more from links
and bookmarks:
- `FPDFAction_GetURIPath`, `FPDFAction_GetFilePath` — distinguish
  "open URI" from "open local file" link actions.
- `FPDFDest_GetView`, `FPDFDest_GetLocationInPage` — the precise
  view-state + viewport-position a named destination resolves to.

The current readers report the action / view type by name (via
the Phase 9 enum helpers) but don't dig into per-action parameters.
**Action**: wrap these in v0.2.0 — they're zero-risk reader
additions.

### Text low-level (rectangle + matrix reads)

`FPDFText_CountRects`, `FPDFText_GetRect`, `FPDFText_GetBoundedText`,
`FPDFText_GetMatrix`, `FPDFText_GetCharAngle`,
`FPDFText_GetFontWeight` — would extend `pdf_text_runs()` /
`pdf_text_chars()` with per-character transformation matrices,
text-flow rectangles, and explicit angles. The current readers
expose bounds + font name + size; these would add geometry
fidelity for table-extraction or text-reflow downstream tooling.

### Annotation surface gaps

`fpdf_annot.h` has ~24 unwrapped symbols. The ones worth shipping
in v0.2.0:
- `FPDFAnnot_AddInkStroke` + `FPDFAnnot_RemoveInkList` — the ink
  annotation surface (we ship `pdf_annot_ink_paths()` as a reader
  but no writer).
- `FPDFAnnot_AppendObject` + `FPDFAnnot_RemoveObject` +
  `FPDFAnnot_UpdateObject` + `FPDFAnnot_GetObject` — the
  page-object-inside-annot surface (stamp annotations primarily).
- `FPDFAnnot_SetURI` — write `/URI` on link annotations. The
  current annotation authoring (Phase 6) covers freetext / square /
  circle / highlight etc. but not URI on link annots specifically.
- `FPDFAnnot_SetAP` — write the appearance stream directly. We
  currently re-touch the rect via `cpp_annot_touch_ap` so PDFium
  regenerates AP automatically; `SetAP` lets advanced callers
  override it explicitly.
- `FPDFAnnot_AddFileAttachment`, `FPDFAnnot_GetFileAttachment` —
  the file-attachment annotation subtype; current Phase 6 surface
  creates a `"fileattachment"` annot but doesn't link a file to it.

### Image authoring

`FPDFImageObj_LoadJpegFile`, `FPDFImageObj_SetBitmap`,
`FPDFImageObj_SetMatrix` — Phase 5 deliberately deferred image
creation pending an FPDF_BITMAP plumbing design. v0.2.0 should
revisit: an `FPDF_BITMAP` constructor (`FPDFBitmap_Create*`) plus
these three setters would round out the v0.1.0 page-object
creation set (`pdf_path_new`, `pdf_rect_new`, `pdf_text_new` →
`pdf_image_new`).

### Clip path authoring

`FPDF_CreateClipPath`, `FPDF_DestroyClipPath`,
`FPDFPage_InsertClipPath`, `FPDFPageObj_TransformClipPath`,
`FPDFPage_TransFormWithClip` — currently we read clip paths
(`pdf_obj_clip_path()`, `pdf_clip_path_segments()`) but can't
create or mutate them.

## Lower-priority gaps

These have legitimate use cases but require deliberate design or
have low cross-user demand.

### Form-fill event callbacks (`fpdf_formfill.h`)

38 unwrapped symbols, mostly `FORM_On*` event handlers
(OnKeyDown, OnLButtonDown, OnFocus, etc.). These would let an R
front-end behave as an interactive form viewer; out of scope for
a non-GUI library. Skip indefinitely unless someone builds a
shiny/htmlwidgets wrapper that needs them.

### Progressive / streaming APIs

`fpdf_dataavail.h` (8 symbols) and `fpdf_progressive.h` (4
symbols) — for asynchronous document loading and threaded
rendering. Useful only when R needs to render a huge PDF without
blocking; current users are batch-mode and don't need it.

### System font integration (`fpdf_sysfontinfo.h`)

7 unwrapped symbols. PDFium will auto-discover the platform fonts
on each OS; these helpers are for embedders that want to override
the font lookup. Skip unless we hit a "missing glyph" bug that
needs platform-specific font installation.

## Refresh procedure

```sh
# Total exported symbols across all bundled headers.
grep -rh "^FPDF_EXPORT" inst/include/fpdf_*.h | grep -oE 'FPDF[A-Za-z_]+' \
  | sort -u | wc -l

# Symbols referenced anywhere in src/.
grep -rh 'FPDF[A-Za-z_]\+' src/*.cpp | grep -oE 'FPDF[A-Za-z_]+' \
  | sort -u | wc -l
```

Run after every PDFium pin bump (`tools/pdfium-version.txt`) and
update the date + provenance block above.

## Caveats

- The audit counts a symbol as "wrapped" when its name appears in
  any `src/*.cpp` file. False positives are possible: a symbol may
  be referenced in a comment without being actually called.
- A symbol called by another shim (e.g. `FPDFAnnot_GetRect` called
  inside `cpp_annot_touch_ap`) counts as wrapped even though we
  don't surface it as a standalone reader. Right call for an
  *API coverage* audit; refresh if you want a *user-surface
  coverage* audit, which would be a different number.
- Symbols that don't appear in the headers but are reachable
  through `FPDF_FORMFILLINFO` callback structs (e.g. the
  `Form_*` event entries) aren't counted as exports; they're
  implementation hooks, not public API.
